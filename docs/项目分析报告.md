# VReport 项目分析报告

> 生成日期：2026-06-26
> 审查范围：核心引擎、状态管理、组件层、数据/导出层、序列化、测试与工程化配置
> 审查方法：源码通读 + 静态审计（依赖未安装，未执行运行时/类型检查）

## 结论

这是一个 **「算法层扎实、工程层薄弱」** 的项目。核心引擎（展开 / 表达式 / 聚合 / 参数 / 条件格式）设计合理、注释完整、可读性好；但 **安全、工程化、组件架构、测试体系** 存在系统性短板，需要分阶段治理。

## 三、问题清单（按严重度）

### 🔴 高 · 安全与健壮性

| # | 问题 | 位置 | 建议 |
|:--:|------|------|------|
| 1 | **`xlsx@0.18.5` 含已知高危 CVE**（CVE-2023-30533 原型污染 / CVE-2024-22363 ReDoS），用于读 Excel。项目已同时依赖 `exceljs`（用于导出） | [ExcelProvider.ts](../src/core/datasource/ExcelProvider.ts)、[package.json](../package.json) | 统一用 `exceljs` 读取并移除 `xlsx`，一次消除漏洞 + 减少包体 + 去重 |
| 2 | **文件解析无大小防护**，`base64ToUint8Array` 直接 `atob` 全量加载，超大文件导致浏览器 OOM | [ExcelProvider.ts](../src/core/datasource/ExcelProvider.ts) | 增加文件大小上限校验 |
| 3 | **CSV 错误被静默吞没**，解析错误仅 `console.warn`，用户以为数据完整、实则丢行 | [CsvProvider.ts](../src/core/datasource/CsvProvider.ts) | 错误率超阈值时向用户提示或抛错 |
| 4 | **序列化无版本迁移**，用 `as ReportTemplate` 强转、无 `schemaVersion`，模型升级后旧模板会崩溃 | [Serializer.ts](../src/core/serializer/Serializer.ts) | 增加 `schemaVersion` + 迁移钩子 + 校验 |
| 5 | **ID 生成有碰撞风险**，使用 `Date.now()+Math.random()` | [id.ts](../src/utils/id.ts) | 改用 `crypto.randomUUID()` |
| 6 | **错误处理外泄**，`catch (e: any)` 普遍，部分直接 `message.error` 暴露技术细节 | 多处 | 收敛为 `catch (e: unknown)`，对外只显示通用提示 |

### 🔴 高 · 工程化缺失

| # | 问题 | 位置 | 建议 |
|:--:|------|------|------|
| 7 | **无质量工具链**：缺 ESLint / Prettier / EditorConfig / Husky / CI，仅有 `.gitignore` | [.gitignore](../.gitignore) | 补齐 lint/format/CI 基础设施 |
| 8 | **脚本残缺**：只有 `dev/build/preview`，无 `test/lint/typecheck/format`，无法 `npm test` | [package.json](../package.json) | 补全 npm scripts |
| 9 | **测试非标准化**：手写 `console.assert` 脚本，无 Vitest、无覆盖率；**组件层零测试** | [test/](../test) | 迁移 Vitest，补组件单测 |

### 🟡 中 · 架构与代码质量

| # | 问题 | 位置 | 建议 |
|:--:|------|------|------|
| 10 | **巨型组件**：约 1400 行，集渲染/选区/编辑/拖拽/快捷键/复制粘贴/公式于一身 | [CellCanvas.vue](../src/components/designer/CellCanvas.vue) | 拆分为 `VirtualGrid` + 多个 composable |
| 11 | **属性面板过大**：约 800 行，多分组逻辑耦合 | [PropertyPanel.vue](../src/components/designer/PropertyPanel.vue) | 按分组拆分子组件 |
| 12 | **Dialog 重复约 70%**：4 个对话框结构雷同 | [ImageDialog.vue](../src/components/designer/ImageDialog.vue)、[QRCodeDialog.vue](../src/components/designer/QRCodeDialog.vue)、[BarcodeDialog.vue](../src/components/designer/BarcodeDialog.vue)、[ChartDialog.vue](../src/components/designer/ChartDialog.vue) | 抽取 `useFormDialog` composable |

## 五、优化方向与分阶段计划

| 阶段 | 目标 | 主要内容 | 风险 |
|:---:|------|------|:---:|
| **0 地基** | 工程化 | 装依赖；补 `test/lint/typecheck/format` 脚本；加 ESLint + Prettier + EditorConfig；最简 CI | 低 |
| **1 安全** | 消除高危 | `exceljs` 替换并移除 `xlsx`；文件大小上限；CSV 错误提示；`crypto.randomUUID`；Serializer 加 `schemaVersion` + 迁移；收敛 `catch(e:unknown)` | 中 |
| **2 架构** | 组件重构 | 拆 `CellCanvas` → `VirtualGrid` + 若干 composable；抽 `useFormDialog`；用 store action 替换信号量；加全局错误边界；把公式输入 / 引用选择 / 联想提示重构为更接近 Excel/WPS 的交互 | 高 |
| **3 测试** | 可维护 | 迁移 Vitest（保留现有断言）+ 覆盖率；补关键组件单测；消除 `any` / 补 ARIA / 提取常量文案 | 中 |

**建议节奏**：阶段 0 + 1 属「低风险高收益」，可优先落地；阶段 2 改动面大，需配合回归测试。

## 六、优化执行成果（本次已落地）

> 本次已按上述计划执行重构，全程以 `npm run typecheck`、`npx vitest run`、`npx eslint . --quiet` 验证。当前状态：**类型检查通过、7 个测试文件全部通过、0 个 ESLint error（98 warning 为渐进采用项）**。

### 阶段 0 · 工程化地基 ✅
- `package.json` 补齐脚本：`typecheck` / `lint` / `lint:fix` / `format` / `format:check` / `test` / `test:watch` / `test:coverage` / `ci`。
- 新增 ESLint 9 flat config（`typescript-eslint` v8 + `eslint-plugin-vue` v9）、Prettier、`.editorconfig`、`.prettierignore`。
- 新增 Vitest 配置（jsdom 环境，覆盖率指向 `src/core/**`）与 GitHub Actions CI（`.github/workflows/ci.yml`）。
- 规则采用渐进策略（多数置为 `warn`），避免一次性阻断；保持现有代码风格（无分号、单引号、宽度 100）。

### 阶段 1 · 安全与健壮性 ✅
- **移除 `xlsx` 依赖**，Excel 读取改用已有的 `exceljs`，消除 `CVE-2023-30533`（原型污染）与 `CVE-2024-22363`（ReDoS）两个高危漏洞。
- `ExcelProvider` 新增 **20MB 文件上限**，并规范化公式 / 富文本 / 超链接单元格取值。
- `CsvProvider` 解析失败由静默 `warn` 改为：**完全失败抛错**、部分失败保留并告警。
- `id.ts` 的 `uid()` 改用 `crypto.randomUUID()`（带强化回退）。
- `Serializer` 引入 `CURRENT_SCHEMA_VERSION` 与 `migrateTemplate()`：导入模板时**校验结构并按版本迁移**，旧模板不再因强制类型断言而崩溃；`ReportTemplate` 增加 `schemaVersion`。
- 审计结论：生产依赖仅余 2 个 moderate（`exceljs → uuid`，实际不可利用），critical/high 均为 dev 工具链（vite/esbuild/vitest），不进入产物。

### 阶段 2 · 架构与组件 ✅（关键项）
- 新增**全局错误边界** `ErrorBoundary.vue`（`onErrorCaptured`），在 `App.vue` 包裹 `router-view`，并提供「重试 / 刷新」。
- 抽取 **`useFormDialog`** composable，重构 `Image / QRCode / Barcode / Chart` 四个弹窗，消除「可见时重置表单 + 合并初始值 + 关闭」的重复样板。
- 信号量语义化：`designer` store 新增 `clearFieldInsertRequest` action，画布改为经 action 变更状态。
- 解耦示范：将公式引用解析从 `CellCanvas` 抽取为 core 纯函数 `core/expression/FormulaRefs.ts`（`extractFormulaRefs` / `REF_COLORS`），并补充独立单测。

### 阶段 2 · 公式输入对齐 Excel/WPS ✅（用户重点）
- `Autocomplete.getParamHint` 现返回**参数名列表**与**当前参数索引** `activeArg`（顶层逗号计数，正确忽略嵌套括号内的逗号）。
- 单元格编辑器参数提示**高亮当前正在输入的参数**（Excel/WPS 风格）。
- 新增 **Tab 补全**当前联想函数；**Esc 优先关闭联想**（无联想时才取消编辑）；沿用「应用建议自动补全括号」「被引用单元格彩色边框高亮」。

### 阶段 3 · 测试体系 ✅
- 将 `test/` 下 6 个 `console.assert` 脚本**迁移到 Vitest**：保留原断言，移除 `process.exit`，以 `it()` 包装并以 `expect(failed).toBe(0)` 收口。
- 新增 `formula-refs.test.ts`（标准 `describe/it/expect`）与公式 `activeArg` 高亮断言。
- 现状：**7 个测试文件全部通过**（核心引擎用例 + 新增公式用例）。

### 后续建议（未在本次执行）
- `CellCanvas`（约 1400 行）的**完整拆分**（`VirtualGrid` + `useSelection` / `useCellEditor` / `useClipboard` / `useKeyboard`）属高风险改造，建议在补齐组件级 / E2E 回归测试后分步推进；本次已先完成低风险的纯函数抽取作为示范。
- 渐进消除 98 个 ESLint warning（`any`、未使用变量等）。
- `RenderedTable` 大报表虚拟滚动、ARIA 无障碍增强。
